Configuring your WordWeaver

This guide assumes you have followed the steps of both installing and setting up WordWeaver. You are therefore assumed to have some knowledge and understanding of Git, Command Line, Python, and Docker. In order to deploy to a production environment, it's also recommended that you have some familiarity with nginx. All of these tools are used in this guide

Before we dig in, have a look at the directory tree below. Please refer to it throughout this guide.

When deploying just to a development environment (the default) there is only one service being composed by docker, the backend. To deploy to production, we also need a certbot service and a reverse proxy.

Backend (WordWeaver)#

This is the main configurable service. It is the wordweaver Python package and API.

Every WordWeaver has its data separated into 4 main sections: Verbs, Pronouns, Options, and Conjugations. This data must be in JSON format. To add your data, create a new folder in src/wordweaver/data named after your languages’s ISO 639-3 code. You will need to add 6 files: the described JSON data files, a blank init.py file, and a models file. The following sections describe the data and models files.

Models#

Your models.py file is found at src/wordweaver/data/<yourlang>/models.py and should contain a model for each data file (Verbs, Pronouns, Options, and Conjugations). If your data does vary from the default that’s been described, you will have to update the associated model.

For example, if you wanted to include a "gender" attribute to your "pronoun" data, you would have to up the pronoun model as follows

class Pronoun(BaseModel):
''' Required '''
gender: str = ''
tag: str = ''

If your data does not vary from the described defaults, the include default models.py file will contain all sufficient information and can be left untouched.

Verbs#

You must have data related to the verb roots available in your WordWeaver in a single verbs.json file. Your verbs should be a JSON array of objects. Minimally, each verb in your verbs array must have a uniquetag, and classes keys.

Default Definition:

class Verb(BaseModel):
''' Required '''
tag: str = ''
classes: List[str] = []

tag#

The tag is the unique identifier for that verb. It must also be URL friendly, otherwise known as a slug. To be URL friendly, a slug must be restricted to certain characters

classes#

The classes key is an array of strings that are CSS class names to apply to that verb. Please visit the WordWeaver UI style customization guide for more information.

Pronouns#

You must have data related to the pronouns available in your WordWeaver in a single pronouns.json file.

Default Definition:

class Pronoun(BaseModel):
''' Required '''
tag: str = ''

tag#

The tag is the unique identifier for that pronoun. It must also be a slug.

Options#

The 'Options' category is a catch-all category for all other types of information associated with choosing a conjugation. You therefore must have all other options related to your data available in a single options.json file.

Default Definition:

class Option(BaseModel):
''' Required '''
tag: str = ''
type: str = ''

tag#

The tag is the unique identifier for that option. It must also be a slug.

type#

The type allows for categorization of options (ex. tense, aspect etc)

Conjugations#

You must have data related to the conjugations available in your WordWeaver in a single conjugations.json.

Each conjugation consists of an input and an output:

class ResponseObject(BaseModel):
input: ConjugationInput
output: Conjugation

The input is an object that includes the tag values for each related verb, pronoun, and option:

class ConjugationInput(BaseModel):
root: str
option: str
agent: str

The output is a list of objects containing response morphemes:

Conjugation = List[ResponseMorpheme]
class ResponseMorpheme(BaseModel):
position: Optional[int]
value: Optional[str]
gloss: Optional[str]
english: Optional[str]
type: Optional[List[str]]

Response morphemes are units that can be concatenated into Tiers. Tiers are defined in the WordWeaver UI.

Edit CORS rules#

Cross-origin resource sharing (CORS) rules are defined in src/wordweaver/main.py. You can edit allowable origins, methods and headers here.

This is a security feature that is intended to make sure that your wordweaver only loads resources (code and scripts) from trusted domains.

Default available routes#

WordWeaver creates an API using your data and exposes it using the following default routes: options, pronouns, verbs, and conjugations. All of these routes are prefixed by /api/v1 by default as well as the chosen domain. By default, if you run the development version, this is localhost. So, once WordWeaver is started, all of your pronouns are available at localhost/api/v1/pronouns.

Reverse Proxy (nginx)#

This is only required for deploying to production environments

The default nginx configuration is very basic. It is configured to work with your Let's Encrypt SSL certification, and it is also configured to force redirects to https.

The configuration is defined as a template volumes/reverse/nginx/conf/wordweaver.template that is formatted with your domain name as defined in .env

Certbot#

This is only required for deploying to production environments

Certbot manages your SSL cert. We recommend using Let's Encrypt. After securing your server ensuring Docker is available, and cloning your repo onto your server, do the following steps to set up automated renewals of your SSL cert.

  1. Change the email and domain in .env
  2. On your server, run init-letsencrypt.sh to get an SSL cert for your site
  3. Spin up your WordWeaver by running docker-compose -f docker-compose.prod.yml up
important

Your DNS records will have to be set up to point to your server
